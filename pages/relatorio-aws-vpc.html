Task 1: Governança e Padronização de Rotas

Objetivo: Identificar e etiquetar corretamente os recursos de rede para conformidade.

1 Localização da Tabela de Rotas

Onde ir: VPC > Tabelas de rotas.

Ação Realizada: Localizada a tabela rtb-0ad4272549e9fe8cf associada à sub-rede privada e alterada a tag Name de "changeme" para "Private".

Explicação Técnica: A padronização de nomenclatura é fundamental para automação e auditoria. Garante que o tráfego da sub-rede privada seja gerenciado por uma tabela específica.

Figura 1: Tabela de rotas rtb-0ad4272549e9fe8cf renomeada para "Private"
Resultado: Governança estabelecida com nomenclatura padronizada para fácil identificação e auditoria.

Task 2: Configuração do Perímetro de Borda (ProxySG2)

Objetivo: Habilitar o acesso público ao segundo nó de Proxy para garantir alta disponibilidade.

1 Liberação da Porta HTTP

Onde ir: EC2 > Grupos de Segurança > ProxySG2.

Ação Realizada: Adicionada regra de entrada para protocolo HTTP (porta 80) vinda de 0.0.0.0/0.

Explicação Técnica: Define a camada de borda como funcional, permitindo que requisições externas cheguem aos servidores de Proxy antes de serem encaminhadas internamente.

Figura 2: ProxySG2 - Regra de entrada HTTP liberada para 0.0.0.0/0
Figura 3: Análise de rota da sub-rede pública - tráfego direcionado ao Internet Gateway
Resultado: Perímetro de borda configurado com alta disponibilidade através do segundo nó de Proxy.

Task 3 & 4: Isolamento da Camada de Aplicação

Objetivo: Remover a exposição direta do servidor de aplicação à internet.

1 Exclusão da Regra Pública

Onde ir: EC2 > Grupos de Segurança > AppServerSG.

Ação Realizada: Exclusão da regra de entrada HTTP para 0.0.0.0/0.

Explicação Técnica: Aplicação do princípio de "Zero Trust" externo. O AppServer passa a ser invisível para varreduras públicas, permitindo acesso apenas via tráfego controlado.

Figura 4: AppServerSG - Regra HTTP pública removida com sucesso
Princípio aplicado: Zero Trust externo - o AppServer agora é invisível para varreduras públicas.

Task 5: Relação de Confiança (Security Group Chaining)

Objetivo: Permitir comunicação apenas entre camadas autorizadas (Proxy -> App).

1 Configuração da Origem por Security Group

Onde ir: AppServerSG > Editar regras de entrada.

Ação Realizada: Adicionada regra HTTP (80) tendo como Origem o ID do Security Group dos Proxies (sg-0c6b5965071a066a8).

Explicação Técnica: Em vez de liberar IPs, liberamos o "grupo". Isso cria um funil de segurança dinâmico onde apenas instâncias com a identidade de "Proxy" podem falar com o AppServer.

Figura 5: Validação da referência ao Security Group dos Proxies
Figura 6: AppServerSG - Regras finais aplicadas com origem nos Security Groups dos Proxies
Conceito chave: Security Group Chaining permite uma relação de confiança dinâmica - novos proxies adicionados ao grupo automaticamente ganham acesso.

Task 7: Acesso Administrativo via Bastion Host (Jump Box)

Objetivo: Acessar o terminal de instâncias privadas de forma segura.

1 Configuração do Bastion Host

Ação Realizada: Renomeação da instância para Bastion e criação do BastionSG permitindo SSH (22). Realizado o salto SSH usando Agent Forwarding.

Figura 7: AppServerSG - Regra SSH liberada apenas para o Security Group do Bastion
2 Comandos de Acesso

No Terminal: Executado ssh -A ec2-user@BastionIP e depois ssh ec2-user@AppServerIP.

Comando de Prova: touch newfile.txt criado no diretório raiz do AppServer.

Figura 8: Terminal - Acesso SSH bem-sucedido ao AppServer via Bastion Host
Explicação Técnica: O Bastion Host atua como um portão controlado. O uso de Agent Forwarding evita o armazenamento de chaves .pem dentro de servidores públicos, uma melhor prática de segurança.

Task 8: Gestão de Sistemas via AWS Session Manager

Objetivo: Gerenciar servidores sem necessidade de abrir portas de rede (Portless Access).

1 Acesso via Session Manager

Onde ir: Console EC2 > AppServer > Conectar > Session Manager.

Ação Realizada: Alteração do arquivo index.html via terminal web com o comando sudo sed.

Explicação Técnica: O Session Manager utiliza o agente SSM e permissões IAM para prover acesso ao terminal via HTTPS (443), eliminando a necessidade de gerenciar chaves SSH ou abrir a porta 22 na rede.

Figura 9: Session Manager - Edição bem-sucedida do arquivo index.html via terminal web
Resultado: Acesso Portless implementado - gerenciamento do servidor sem abrir portas de rede.
